Pattern 1
Single Agent — Reasoning Only
Reasoning
R0 / R1
Session-scoped · Low latency · Simple
🧠 A single Capability running a bounded Reasoning scope. One Strands agent hosted in AgentCore — perceives the request, reasons over available tools, acts. Scoped to a session. No delegation to other Capabilities, no durable process state. Typical for R0 retrieval and R1 operational write Capabilities.

Adding Coordination scope
When a Capability needs to delegate to other Capabilities — not just call tools — declare the Coordination scope. The LLM decides which specialist to invoke (agents-as-tools) or a graph defines the topology (LangGraph). Still session-scoped. No durable state.
Pattern 2
Orchestrator + Specialists — Reasoning + Coordination
Reasoning Coordination
R0 / R1 typical
Dynamic delegation · Session-scoped
🎯 An orchestrator Capability delegates to specialist Capabilities via agents-as-tools. The LLM decides dynamically which specialist to invoke and in what order. Each specialist is itself a Reasoning-scope Capability with its own Published Definition, Safety Envelope, and Risk Tier. All are hosted in AgentCore.

When Process scope is required
Any Capability declared R2 or R3 must declare the Process scope and provide a Workflow Contract. The Registry will reject publication without it (I4). Process scope adds durable execution, Workflow Contract enforcement, and fail-closed audit semantics — not optional for regulated or customer-adjacent writes.
Pattern 3 ⭐
Durable Workflow — Reasoning + Coordination + Process
Reasoning Coordination Process
R2 / R3
Mandatory R2/R3 · Durable · Compensatable
⭐ Step Functions implements the Workflow Contract — deterministic outer structure with durable state, compensation, and fail-closed audit. AgentCore-hosted Strands agents provide dynamic Reasoning within each step, with Coordination scope available within steps. Mandatory for any Capability that writes customer-adjacent or regulated state.
Workflow Contract (I4)
This pattern requires a Workflow Contract declared in the Capability Definition. The Contract defines: correlation ID standard, idempotency requirements, compensation semantics for partial execution, ABAC enforcement at each Step Functions task boundary, replay guarantees. Registry rejects R2/R3 publication without it. Workflow Contract may be authored independently by compliance/process design function via Workflow Studio and referenced by the Capability Definition.
Invariants in effect
I1 Published Definition required
I2 Safety Envelope + Risk Tier validated at publish
I3 Identity Context propagates through Step Functions input to every Lambda/AgentCore task
I4 Workflow Contract declared and validated — R2/R3 mandatory
I5 Audit Record write-ahead — committed before state-changing tool calls at R3. R2: fail-closed on pipeline outage.
I6 Safety Envelope immutable — Guardrails enforces in each AgentCore task

Process
Step Functions Orchestrator
  • Implements the Workflow Contract declared in the Capability Definition
  • Provides: sequence, Parallel states, Map iterations, Retry with exponential backoff, Catch, timeouts
  • Correlation ID flows as execution name — traceable end-to-end
  • Compensation state defined in ASL — if any step fails, compensating actions execute
  • Idempotency key checked before state-changing steps — safe to replay
Workflow Contract fields
correlation_id: {$.execution_id}
idempotency_key: {$.request_hash}
compensation: compensate_state
abac_boundary: per-task
AgentCore Task (Reasoning)
  • Each Step Functions task invokes AgentCore-hosted Strands agent
  • Agent reasons dynamically within the bounded scope of its task
  • Can use agents-as-tools internally for Coordination within the step
  • IAM execution role inherits session tags from Step Functions input — Identity Context maintained (I3)
  • Guardrails enforces Safety Envelope on every LLM call within the task (I6)

HITL Gate · Risk Tier-driven
  • waitForTaskToken pauses execution — Step Functions waits for human callback
  • HITL Service routes to designated reviewer via SNS
  • Threshold and conditions defined in Capability Definition HITL Conditions field
  • R3: HITL cannot be waived — mandatory before state-changing execution continues
  • R2: HITL threshold configurable but required at declaration time

MCP Tool Gateway
  • All tool calls route via MCP Tool Gateway — agents cannot call tools directly (I3)
  • RFC 8693 token exchange per MCP server — scoped credentials, PrivateLink transit
  • ABAC enforcement at each tool boundary — tenant isolation structurally enforced
  • R3: write-ahead audit commit before any state-changing tool call executes
  • R3 audit pipeline outage: execution halts — no exception path

Audit Trail · R2/R3
  • Audit Record written synchronously at each step (I5) — fail-closed at R2/R3
  • S3 Object Lock Compliance mode — retention driven by Risk Tier, immutable
  • Step Functions console: full execution graph, state history, compensation path
  • Langfuse: nested agent traces, tool spans, LLM-as-judge per task
  • CloudWatch: Lambda metrics, AgentCore latency, error rates per step
