Central Artefact
Capability Definition
The immutable, versioned specification of an AI capability. Every authoring path — IDE, Langflow, Guided Assembly, Workflow Studio — produces exactly one. Once Published, no field changes; a new version is created. All governance is a function of this artefact. The Registry stores and enforces it. The Execution is a runtime instance of it.

Safety Envelope
Versioned, provider-agnostic policy attached to a Capability Definition. Declares permitted content scope, PII handling posture, prompt injection controls, output constraints. Evaluated at publish time and immutable on the published version. The Guardrails Service is its runtime enforcement mechanism — the Envelope is the declared intent, Guardrails is the mechanism. A wrong Guardrails configuration is a platform fault; a wrong Safety Envelope is a governance failure.
Risk Tier
Classification of what a Capability is permitted to affect. Declared at definition time, validated by the Registry at publish. Drives proportionality across the entire governance stack: evaluation rigour, HITL threshold, audit retention period, Workflow Contract requirement, fail-open vs fail-closed audit semantics. R0 is structurally enforced — the Registry inspects Tool Bindings. R1–R3 are declared by the team and validated through evaluation.
Execution Model
A declared field on the Capability Definition — not an implementation choice made at build time and hidden in code. Three composable scopes: Reasoning (always present, platform standard), Coordination (how this Capability delegates to others), Process (durable execution across invocations — mandatory for R2/R3). The Registry can reason about execution model because it's declared, not inferred.

Runtime Instance
Execution
A runtime instance of a Published Capability Definition processing a single request. Traces exactly to one Definition version, one Identity Context, and produces one Audit Record. Cannot exist without all three. R2/R3 Executions additionally trace to a Workflow Contract. An Execution is the atomic unit of platform accountability — every governance claim the platform makes is ultimately a claim about what happened in an Execution.
Identity Context
The propagated security principal for an Execution. Established once at session start via OIDC → IAM session tags (tenant_id, brand, role, use-case). Immutable for the full duration of the Execution — raw user credentials stop at the Lambda Authorizer. Every tool invocation, ABAC decision, and audit record is a function of it. For delegated Executions (Capability calling Capability), subject identity stays stable; delegation claims (acting_as_capability_id, invocation_chain) are added.
Audit Record
The immutable output of every Execution. Contains: Identity Context, Capability Definition version, Safety Envelope version in effect, Risk Tier in effect, all MCP tool calls and results, guardrail input/output traces, token counts, cost attribution. Written synchronously — not best-effort logging. Failure to write is a platform fault, not a degraded state. Retention period driven by Risk Tier. R0/R1 fail-open on audit pipeline outage. R2/R3 fail-closed — no state-changing tool calls without audit commit.
Workflow Contract
A formal specification required for all R2/R3 Capability Definitions. The Registry rejects R2/R3 publication without one — this is invariant I4, not a convention. Defines: correlation ID standard, idempotency requirements, compensation semantics for partial execution, ABAC enforcement at each step boundary, replay guarantees. May be versioned and owned independently by a compliance or process design function, then referenced by Capability Definitions.
Invariants
What Must Hold Unconditionally
Six rules enforced by the platform, not aspirationally observed. They are what make governance provable rather than claimed. Each maps to a specific enforcement point in the architecture.
I1
No Execution without a Published Definition
Draft or Review-state Capability Definitions cannot execute in production. The Agent Runtime checks lifecycle state before accepting an invocation request.
Enforced by: Agent Runtime · Capability Registry
I2
No publish without Safety Envelope and Risk Tier
A Capability Definition cannot transition to Published without a validated Safety Envelope and a declared Risk Tier. The Registry rejects the state transition.
Enforced by: Capability Registry publish API
I3
No tool invocation outside Identity Context
Every tool call must occur within the scope of the Identity Context established at session start. RFC 8693 token exchange enforces this at every MCP boundary.
Enforced by: MCP Tool Gateway · RFC 8693
I4
R2/R3 requires a Workflow Contract
Every R2 or R3 Capability Definition must declare a Workflow Contract. The Registry rejects publication without one. R0/R1 are exempt.
Enforced by: Capability Registry publish API
I5
Every Execution produces an Audit Record
The Audit Record is synchronous to the Execution — not best-effort background logging. R2/R3: write-ahead before state-changing tool calls. R0/R1: fail-open with alerting.
Enforced by: Audit & Compliance service
I6
Safety Envelope is immutable on publish
The Safety Envelope is evaluated at publish time and cannot be modified on a published version. Guardrails enforce it at runtime but cannot change it. Changes require a new version.
Enforced by: Capability Registry · Guardrails Service
Risk Tier
Governance Proportionality
Risk Tier is the governance driver. Every control in the platform — evaluation rigour, HITL threshold, audit retention, Workflow Contract requirement, fail-open vs fail-closed — is derived from the Risk Tier declared in the Capability Definition. The tier must be declared by the authoring team and is validated, not inferred.
R0
Read-Only
No write tool bindings. Retrieves from approved knowledge sources and returns a response. Cannot write state anywhere regardless of domain.
Write permissionsNone — structurally enforced
Workflow ContractNot required
HITL thresholdNot required
Audit retentionStandard
Audit pipeline outageFail-open with alerting
Evaluation gateFunctional quality
R0 is the only tier structurally enforced — the Registry inspects Tool Bindings at publish and rejects any write-capable binding. A team cannot accidentally publish an R0 Capability that writes state.
R1
Operational Write
Writes to internal operational systems only. Subject of every write must be non-customer. Target system must carry no regulatory reporting obligation.
Write permissionsInternal operational only
Workflow ContractNot required
HITL thresholdOptional · team-defined
Audit retentionStandard
Audit pipeline outageFail-open with alerting
Evaluation gateFunctional + scope confirmation
R1 is declared, not structurally enforced. The evaluation gate must include explicit confirmation that no customer data is in scope — misclassification of R2 work as R1 should be a deliberate act, not an oversight.
R2
Customer-Adjacent
Writes to systems where the subject is a customer or data carries regulatory significance — account preferences, communication opt-outs, KYC flags, CRM state, RG monitoring inputs.
Write permissionsCustomer-adjacent systems
Workflow ContractMandatory — I4
HITL thresholdRequired · threshold defined
Audit retentionExtended
Audit pipeline outageFail-closed
Evaluation gateExtended suite required
The R1→R2 boundary is the critical classification decision. Test: if this write is wrong, could a customer be harmed or a regulator ask questions? If yes, R2 minimum.
R3
Regulated
Writes to systems with a direct regulatory obligation — Responsible Gambling interventions, AML flags, financial transactions, regulatory disclosures. Wrong writes are reportable events.
Write permissionsRegulated systems only
Workflow ContractMandatory + compensation — I4
HITL thresholdMandatory · cannot be waived
Audit retentionMaximum · S3 Object Lock
Audit pipeline outageFail-closed · write-ahead
Evaluation gateMaximum rigour
R3 audit is write-ahead — the Audit Record is committed before any state-changing tool call executes. A failed audit write at R3 halts execution. No exception path.
Execution Model
Declared Scopes, Not a Framework Pick-list
The Execution Model is a declared field on the Capability Definition — visible to the Registry, reasoned about by governance. Three composable scopes, not three alternative frameworks. A Capability declares which scopes apply; the platform enforces the governance consequences of that declaration.
Reasoning
Reasoning
The agent's internal ReAct loop — perceive, reason, act, observe, repeat. Every Capability uses this scope. Scoped to a single agent invocation processing a single request. The fundamental execution unit.
Implementation: AWS Strands — LLM-driven tool selection and invocation. Platform standard. Not configurable away.
Always present · platform standard
Coordination
Coordination
How this Capability delegates work to other Capabilities. Scoped beyond the single agent to a topology of agents. Optional — declare only when the Capability genuinely orchestrates others. Two patterns available.
LLM-driven: Strands agent-as-tool — the orchestrator LLM decides dynamically which specialist to call.
Graph-driven: LangGraph — deterministic or conditional topology defined explicitly in code.
Optional · declare when orchestrating other Capabilities
Process
Process
Durable execution spanning multiple agent invocations. Scoped beyond a session. Mandatory for R2/R3 — any Capability touching regulated or customer-adjacent state must be able to prove ordered, compensatable, auditable execution across invocations.
Implementation: AWS Step Functions — durable state machines with defined retry, timeout, parallel, and compensation semantics. Correlation ID and idempotency enforced at the state machine level.
Mandatory for R2/R3 · optional otherwise
