Observability & Audit
Five always-on signal layers covering every Execution — Langfuse traces, CloudWatch metrics, CloudTrail IAM audit, platform Audit Records (I5), and Step Functions execution history (R2/R3). What fires, what's in it, where to look first, and how cost flows to tenants.
Section 1
The Observability Stack
What fires · on every Execution
🔭
Langfuse
Trace · Spans · LLM-as-judge
Always on
  • Full nested trace tree per Execution — root span, agent loop spans, tool call spans, LLM call spans
  • LiteLLM callback fires automatically on every model call — no instrumentation required per Capability
  • Tool call spans include: tool name, input parameters, response status, latency
  • LLM-as-judge evals configurable per Capability — automatic quality scoring on sampled outputs
  • For Coordination scope: orchestrator span wraps specialist spans — full delegation chain visible
Retention: 90 days default
Query: Langfuse UI / API
📊
CloudWatch
Metrics · Alarms · Dashboards
Always on
  • LLM Gateway: request rate, token throughput, cache hit rate, circuit breaker state, cost per tenant/capability
  • AgentCore: agent invocation count, Execution latency p50/p95/p99, error rate, timeout rate
  • Guardrails: policy violation count by type (PII, injection, denied topic), block rate
  • MCP Gateway: tool call volume, RFC 8693 exchange latency, downstream error rate
  • Alarms feed SNS → PagerDuty for production alerting thresholds
Retention: 15 months
Query: CW dashboards / Metrics Insights
🗒️
CloudTrail
IAM · API · Compliance log
Always on
  • Records every IAM AssumeRole and session tag assignment — proof of Identity Context establishment
  • All Registry API calls: publish events, lifecycle transitions, ABAC check results
  • S3 Object Lock put/get events on Audit Records — tamper-evidence chain
  • Guardrails policy evaluation events — who, which capability, what outcome
  • All KMS key usage events — decryption of Audit Records auditable independently
Retention: 7 years (S3 Object Lock)
Query: Athena over CloudTrail Lake
🔒
Audit Record
I5 · Platform invariant
Always on · I5
  • Written synchronously on every Execution — not background, not async (I5)
  • Contains: Identity Context, Capability Definition version + SHA-256, Safety Envelope version, all tool calls, guardrail traces, token counts, cost, Risk Tier
  • Semantics scale with Risk Tier — see Section 3 for full anatomy
  • R0/R1: fail-open on pipeline outage — execution continues, alert fires
  • R2/R3: fail-closed — Execution halts if Audit Record cannot be committed
  • R3: write-ahead — committed before any state-changing tool call executes
Retention: Risk Tier-defined
Storage: S3 Object Lock · Compliance mode
🗺️
Step Functions
Workflow history · R2/R3 only
R2 / R3 · Process scope
  • Full execution graph in AWS console — state-by-state visibility, input/output at every transition
  • Compensation path visible when triggered — which states executed, what compensated
  • HITL approval events recorded: reviewer identity, decision, timestamp, task token
  • Correlation ID links Step Functions execution to Langfuse trace and Audit Record — single artefact per Execution
  • Execution history retained 90 days in service; Audit Record in S3 is permanent
Retention: 90 days in SFN
Query: SFN console / GetExecutionHistory API

Cost & quota attribution
Every token consumed is attributed to a tenant / capability_id / session triplet at the LLM Gateway. Cost flows up through quota enforcement, circuit breaker, and spend reporting — giving per-brand cost visibility across the entire Flutter portfolio.
Section 4
Cost & Quota Attribution
LLM Gateway · per-tenant · real-time
Token Attribution
  • LLM Gateway tags every request at entry with tenant_id · capability_id · session_id — cost record opened before model call
  • Prompt cache hits reduce billable input tokens — cache hit rate tracked in CloudWatch per capability, surfaced in Langfuse spans
  • Token counts (input, output, cached) written to Audit Record on Execution close — cost is part of the immutable record
  • Cost attributed in real time — no post-hoc reconciliation needed
  • Multi-model Executions (where LLM routing selects different providers per step): each model call attributed separately
Attribution key
tenant_id: pp-001
capability_id: cs-agent-v3
session_id: sess-uuid · model: claude-sonnet-4
Quota Enforcement & Circuit Breaker
  • Burst limit: checked on every request — requests per minute per tenant/capability. Exceeded → 429, no model call made, no cost incurred
  • Sustained limit: rolling window token budget per tenant — prevents runaway cost from malfunctioning agent loops
  • Hard budget cap: configured per tenant in LLM Gateway. When cumulative spend reaches cap, circuit breaker opens — all requests for that tenant return 429 with `budget_exceeded` error code
  • Circuit breaker state visible in CloudWatch as a binary alarm — on-call notified immediately on open
  • Budget cap is tenant-configurable via Registry API — each brand controls its own ceiling
Circuit breaker states
CLOSED → normal operation
OPEN → budget_exceeded · 429 returned
Reset: manual via Registry API
Spend Reporting & Attribution
  • Per-tenant spend dashboard in CloudWatch: daily/weekly/monthly spend by brand, by capability_id, by model
  • Langfuse cost metadata: each trace carries token counts and estimated cost — enables per-Execution spend visibility for debugging expensive agent loops
  • Prompt caching savings calculated and displayed — shows ROI on cache-eligible system prompts. R0/R1 Capabilities with stable system prompts typically see 70–85% input token reduction
  • Cost data in Audit Records enables compliance-grade spend audit — cost attribution is tamper-evident at R2/R3
  • Exportable to FinOps tooling via CloudWatch Metrics Streams → S3 → Athena query
Spend query example
SELECT capability_id, SUM(cost_usd)
FROM audit_records
WHERE tenant_id='pp-001' AND date='2026-02'
